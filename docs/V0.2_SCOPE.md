## 1. V0.2 应该做什么？

### 1.1 核心目标：额度管理（Quota Management）

V0.1 证明了"新增 Provider 不改 core/"。V0.2 要证明"新增横切关注点不改 core/"。

额度管理是第一个横切关注点（cross-cutting concern），它跨越所有 Provider，但不应该污染 Provider/Bridge 接口。

#### 1.1.1 Quota Manager

- **职责**: 跟踪每个 Provider 的额度消耗，在额度耗尽时通知 Router 降级
- **位置**: `core/quota.py`（新增文件，不改现有文件）
- **接口**:
  - `QuotaManager.register(provider_name, quota_info)`
  - `QuotaManager.consume(provider_name, amount=1)`
  - `QuotaManager.remaining(provider_name) -> int`
  - `QuotaManager.is_available(provider_name) -> bool`
- **与现有架构的关系**:
  - Provider 的 `quota_left()` 已存在（V0.0.6 冻结），QuotaManager 调用它获取初始额度
  - Router 在 `execute()` 前调用 `QuotaManager.is_available()` 做预检查
  - Router 在 `execute()` 后调用 `QuotaManager.consume()` 扣减
  - **Provider 接口不变**，QuotaManager 是 Router 的协作对象

#### 1.1.2 自动降级增强

V0.1 的 Router 已有 fallback 逻辑，但只是"Provider 不可用时找下一个"。V0.2 需要：

- 额度耗尽 → 自动切换到 fallback Provider
- 切换时记录原因（`quota_exhausted` / `provider_unhealthy` / `auth_failed`）
- Result.metadata 增加 `fallback_reason` 字段（**这是 Result 的兼容性扩展，不破坏 Stable 承诺**）

### 1.2 Provider 健康检查集成

- `health()` 已在 Provider 接口中定义（V0.0.6 Stable）
- V0.2 让 Router 真正使用它：执行前检查 health()，失败则降级
- 健康检查结果缓存在 QuotaManager 或 HealthChecker 中（TTL 60s）

### 1.3 CLI 增强

- `ai-hub quota` 命令：查看所有 Provider 的额度状态
- `ai-hub health` 命令：执行健康检查
- 这些是 CLI 层的新增，不改 core/

## 2. V0.2 明确不做什么？

### 2.1 不做 AI 智能路由

- V0.3 的任务
- V0.2 仍然使用关键词匹配（`classify()`）
- **原因**: AI 路由需要 LLM 调用，引入新的复杂度和成本。在额度管理还没搞定之前加 AI 路由，会混在一起调试困难

### 2.2 不做 GUIBridge

- V0.3 的任务
- GUIBridge 涉及平台差异、会话管理、状态同步，复杂度远超 V0.2 范围
- V0.2 保持 GUIBridge 为预留接口（`run()` 返回 not implemented）

### 2.3 不做任务分解

- V0.5 的任务
- 任务分解需要多步执行、中间状态管理、部分失败回滚——这些是 V1.0 Agent 编排的前置能力，不属于 V0.2

### 2.4 不做 Provider 热加载

- V2.0 插件生态的任务
- V0.2 的 Provider 仍然是 `cli/main.py` 中硬编码注册
- **原因**: 热加载需要插件发现机制、版本管理、安全沙箱——这是另一个维度的问题

### 2.5 不做并发执行

- V0.2 的 `router.execute()` 仍然是同步阻塞
- 并发执行涉及任务队列、资源锁、超时取消——留给 V0.5
- **原因**: 在单任务执行链路还没完全稳固之前，并发只会放大问题

### 2.6 不做流式输出

- V0.2 的 Bridge.run() 仍然返回完整结果
- 流式输出需要 Bridge 接口变更（或新增 `stream()` 方法）——这属于 Bridge API 变更，需要 ADR
- **原因**: 流式输出是体验优化，不是功能缺失。V0.2 优先保证额度管理这个核心链路

### 2.7 不做成本核算

- ProviderMetadata 有 cost 字段（预留），但 V0.2 不实现成本计算
- **原因**: 不同 Provider 的计费模型差异巨大（按 token / 按次 / 按时长），统一定价模型需要市场调研

### 2.8 不改 core/ 现有文件

| 文件 | V0.2 是否可改 | 原因 |
|------|-------------|------|
| `core/provider.py` | ❌ | Provider Stable |
| `core/registry.py` | ❌ | CapabilityRegistry Stable |
| `core/result.py` | ❌ | Result Stable（metadata 字段可扩展） |
| `core/task.py` | ❌ | Task Stable |
| `core/capabilities.py` | ❌ | Capability Stable（可新增标签） |
| `core/bridge.py` | ❌ | Bridge Stable（V0.1.1 冻结） |
| `router/router.py` | ⚠️ 可改内部实现 | Router 外部接口 Stable，内部可加 QuotaManager 调用 |
| `cli/main.py` | ✅ 可改 | CLI 不是 Stable API |

**新增文件允许**:
- `core/quota.py`（QuotaManager）
- `core/health.py`（HealthChecker，如果需要）
- `cli/commands/quota.py`（CLI 新命令）

## 4. 设计约束

### 4.1 Frozen Interface 承诺

V0.2 必须遵守的 Stable 承诺：

| 接口 | 承诺来源 | V0.2 行为 |
|------|---------|----------|
| Task | V0.0.6 Stable | 不改 |
| Result | V0.0.6 Stable | 不改（metadata 是 dict，可扩展） |
| Provider | V0.0.6 Stable | 不改 |
| CapabilityRegistry | V0.0.6 Stable | 不改 |
| Capability | V0.0.6 Stable | 可新增标签，不删现有 |
| Router 外部接口 | V0.0.6 Stable | 不改签名 |
| Bridge | V0.1.1 Stable | 不改 |

### 4.2 不推翻现有架构

- 不引入事件总线
- 不引入中间件链
- 不引入依赖注入容器
- 不引入异步框架
- QuotaManager 是一个普通 Python 类，通过构造函数注入 Router

### 4.3 YAGNI 原则

- 不实现"未来可能需要"的 quota 策略（如按 user 隔离、按时间段重置）
- 只实现当前 4 个 Provider 需要的 quota 逻辑
- quota_type 只有 `daily` / `monthly` / `unlimited` / `unknown` 四种（已在 metadata 中定义）

## 6. 风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| QuotaManager 逻辑渗透到 Provider | 中 | 高 | Contract Test 增加 quota 隔离检查 |
| Router 变更引入 regression | 中 | 中 | 现有 18 个测试 + 新增测试 |
| 过度设计 quota 策略 | 高 | 中 | YAGNI 检查（每个 PR 回答必要性问题） |
| Result.metadata 被滥用 | 中 | 低 | 文档约定 metadata key 命名规范 |
