## 总体结论

> ADR-0017：APPROVED（Proposed）
> 我会给 10.0 / 10（作为 ADR，而不是代码实现）。
>
> 原因很简单：
> 它没有急着做 ExecutionStore，而是先把 Execution 的模型定义出来。
> 这是我认为整个 V0.9.x 到目前为止最重要的一次架构决策。

### ChatGPT 关键评价

> 我认为你现在已经把 Runtime 分成了三个世界：
>
> Plan ─ 业务世界
> Execution ─ 运行时世界
> History ─ 存储世界
>
> 这是一个非常重要的分层。
>
> 很多 Runtime 最后会变成：
> Plan → Executor → SQLite → UI → Trace → Metrics（全部混在一起）
>
> 而现在你的方向是：
> Execution → Event → Consumer → Storage/UI/Metrics
>
> 这是事件驱动 Runtime 的标准演进路线。

### 演进路线（V0.8 → V0.10）

```
Provider Runtime (V0.8)
        ↓
Planner Runtime (V0.9.0-0.9.3)
        ↓
Execution Runtime (V0.9.4)   ← 本版本
        ↓
Workflow Runtime (V0.10)
```

> 这条路线的抽象层次是清晰的，各阶段职责也没有明显重叠。

## 逐项反馈

### D1 ExecutionEvent — 10/10 ✅ 建议增加 event_id

> 粒度比较合理。不会太细（不会记录 `provider_request_sent`），也不会太粗（不会只有 `step_finished`）。
>
> 但建议增加 `event_id`（UUID 或递增整数）。原因不是现在，而是未来：
> - Execution History
> - Replay
> - Filtering
>
> 都会方便很多。timestamp 不是唯一键。

**采纳**：在 `ExecutionEvent` 加 `event_id: str`（UUID）。

### D2 EventBus — 10/10 ✅ 保持简单

> 现在不要做复杂。
> 不要提前加入：priority / wildcard / event hierarchy / async / sticky event
> 唯一建议：让接口保持稳定
>
> 例如：`subscribe(event_type, callback)` — 即使 V0.9.4 内部忽略 event_type，以后可以升级。这样不会破坏 API。

**采纳**：保持简单实现，subscribe 接口预留 `event_type` 参数（V0.9.4 内部暂不过滤）。

### D3 ExecutionMetrics — 10/10 ✅ 关键解耦建议

> 我最满意的地方。因为很多项目直接把 latency / token / cost / retry 塞进 ExecutionResult，最后 ExecutionResult 越来越像 Prometheus。
> 你把 ExecutionResult 和 ExecutionMetrics 拆开，这是正确方向。
>
> 我唯一建议：Metrics 尽量保持可测量。
> 保留：latency / token / retry / cost
> 不要放：status / provider / error（这些不是 Metrics）

**采纳**：D3 现状就是只含可测量字段。ADR-0017 已隐含遵守，明示化即可。

### D4 ExecutionRecord — 10/10 ✅ 赞同「不实现只预留」

> 千万不要为了"架构完整"现在就写 300 行 ExecutionRecord。没有 Consumer / Storage / Trace / 没有意义。
> ADR 先定义 Concept，实现以后。非常合理。

**采纳**：保持现状。

### D5 TraceCollector vs PlanStore — 10/10 ✅ 关键架构建议

> 不要统一。统一以后会越来越难维护。
> 原因：
> - PlanStore 回答「发生了什么？」
> - Trace 回答「怎么发生的？」
>
> inspect → PlanStore
> trace → TraceCollector
> 非常合理。
>
> 但是你提出的用户困惑是真实存在的。
> 我的建议不是统一 Store，而是建立关联：
>
> - TraceCollector: 增加 `has(plan_id)`
> - PlanStore: 增加 `trace_available`
> - inspect: 显示 `Trace: Available` / `No Trace`

**采纳**：在 V0.9.4 实现 `TraceCollector.has(plan_id)` + inspect 显示 "Trace: Available/No Trace"。

### D6 trace 视图 — 10/10 ✅ 关键命名建议

> 我很喜欢。但是建议：不要把 trace 理解成日志，而是 Timeline。
>
> 例如：
> 12:01:00 Plan Started
> 12:01:01 Planner Finished
> 12:01:03 Step1 Started
> 12:01:05 Gemini Finished
> 12:01:08 Plan Finished
>
> Timeline 比日志价值更大。

**采纳**：trace 命令名 → "Timeline" 强调。人类可读输出加真实时间戳 + 相对时间（如 `0.0s`, `0.3s`）。

### D7 schema_version — 关键策略建议

> 我支持升级 1 → 2。但是建议：
> **Schema Version 只因为 JSON Contract 变化而变化。不要因为新增字段就升级。**
>
> 例如：如果只是新增 Optional 字段，Schema 未破坏，我不会升级。
> 我建议：只有 Consumer 必须修改，才 Schema Version++。否则 Version Inflation。

> Q2 Forward / Backward：Consumer 永远容忍缺字段。
> Postel's Law：Be conservative in what you send, be liberal in what you accept.

**采纳**：V0.9.4 schema_version 维持 "1"（不升级到 "2"）。
理由：V0.9.4 新增字段（events、execution_metrics、aggregate_metrics）都是 Optional，老 consumer 缺字段时静默忽略即可。
V0.9.4 文档强化 Postel's Law 兼容性原则。

### Q4 latency 计算 — 关键派生 vs 显式记录

> 三种都记录是浪费。建议：
> - Provider latency：显式记录在 `provider_finished.latency_ms`
> - Step latency：派生（`step_finished.timestamp - step_started.timestamp`）
> - Plan latency：派生（`plan_finished.timestamp - plan_started.timestamp`）
>
> 避免重复。

**采纳**：V0.9.4 Event 中只显式记录 Provider latency。Step/Plan latency 由 Consumer 派生（`get_trace` 时计算）。

### Q5 SQLite 跨进程 — 推迟

> 先不支持跨进程。Runtime 现在单进程。
> 为了一个未来可能存在的需求：现在就引入 File Lock / Retry / Busy Timeout / IPC 不值得。
> 等真正有 Daemon 或 Multiple CLI 同时运行的需求再解决。

**采纳**：V0.9.4 仍单进程。V0.9.5 引入 SQLite 时也先单进程，不做跨进程同步。

### Q6 Workflow Runtime 能力优先级（V0.10）

> 我很赞同「改名为 Workflow Runtime」。不过建议把能力重新排序：
>
> 1. **Dependency**（依赖）
> 2. **Conditional**（条件）
> 3. **Retry**（重试）
> 4. **Checkpoint**（检查点）
> 5. **Resume**（恢复）
>
> DAG 只是 Dependency 的一种表示。Workflow 才是真正抽象。

**采纳**：V0.10 路线重写为：Dependency → Conditional → Retry → Checkpoint → Resume。

### Q7 PlanStore 命名 — 维持

> 不要改。PlanStore 名字是正确的。因为它存的是 Plan，不是 Execution。
> 未来 ExecutionStore 应该是另一个东西。
> PlanStore ↓ Plan / ExecutionStore ↓ ExecutionRecord
> 两者职责不同。
> 如果今天把 PlanStore 改名 ExecutionStore。一年以后真正的 ExecutionStore 就没名字了。

**采纳**：PlanStore 维持现有命名。

## ⚠️ 必须加入 ADR 的设计原则

> **ExecutionEvent 是 Runtime 唯一的执行事件来源（single source of execution truth）。**
>
> 所有 Trace、Metrics、History、UI 都应基于 ExecutionEvent 派生，而不是直接读取 Executor 内部状态。
>
> ```
> Executor
>    │
>    ▼
> ExecutionEvent
>    │
>    ├── TraceCollector
>    ├── MetricsCollector
>    ├── SQLite ExecutionStore (V0.9.5+)
>    ├── JSON Exporter
>    └── Future UI
> ```
>
> 以后新增任何观察能力，都不用侵入 Executor。

**采纳**：作为 ADR-0017 的核心设计原则。
