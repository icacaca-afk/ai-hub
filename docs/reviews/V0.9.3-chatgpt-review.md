## 关键建议

### ✅ Q1: schema_version 顶层 vs 嵌套

**采纳**（保持顶层）。理由：

> metadata 将来一定越来越复杂。Schema Version 属于"如何解释整个 JSON"，不是"如何解释 Plan"。这是两个不同层。

**采纳子建议**：纯数字 `"1"`, `"2"`, `"3"`，不要 `v1`, `v2`。
当前已是 `"1"`。

### ✅ Q2: PlanStore 不持久化

**采纳边界**。理由：

> PlanStore + inspect 是一个 Cache，不是 Database。V0.9.3 最正确的边界。

**采纳子建议**：文档明示"运行时缓存，不保证进程退出后仍存在"。
→ 已加 `(Current Process Only)` 标识。

**采纳子建议**：`max_size=10` 提为 `DEFAULT_STORE_SIZE = 10` 常量，避免魔法数字。

### ✅ Q3: CLI Exit Code

**不采纳** `exit 2 = partial`（按 ChatGPT 建议推迟到 V0.10）：

> 目前 Runtime 没有真正意义上的 Partial。
> 真正需要 exit 2 的时候是 V0.10（如 Step1 OK / Step2 OK / Step3 Failed / Step4 Skipped）。

### 🔄 Q4: V0.9.4 方向（按 ChatGPT 优先级重排）

1. **第一优先级** — **Execution Event 模型**（不是 SQLite）
   - Event 类型：Planning Started / Planner Finished / Executor Started / Step Started / Provider Selected / Provider Finished / Step Finished / Plan Finished
   - 定义 `ExecutionEvent` 数据类（type / timestamp / step_id / provider / latency）
   - SQLite / JSON / Memory 都只是 Consumer

2. **第二优先级** — **ExecutionMetrics**（与 ExecutionResult 解耦）
   - 字段：latency_ms / token_in / token_out / cost / retry
   - 单独 `ExecutionMetrics` 数据类，避免 ExecutionResult 越来越胖
   - 留扩展：cache_hit / queue_wait / network_latency / provider_latency

3. **第三优先级** — **Trace 视图**（比 SQLite 更重要）
   - `ai-hub trace <plan_id>` 新命令
   - 时间线展示：Planning → LLM → Rule → Step1 → Gemini → Success → Step2 → Claude → Success

### 🔄 长期建议：ExecutionRecord 抽象（V0.9.4+ 引入）

```python
ExecutionRecord:
    plan       # 计划
    events     # 事件流
    metrics    # 指标
    result     # 结果
```

- `inspect` 实质上就是看 `ExecutionRecord`，不是 `Plan`。
- 从现在开始区分：**Plan**（业务）/ **Execution**（执行过程）/ **History**（历史持久化）。
- 避免 PlanStore 慢慢演变成 ExecutionStore（职责不同）。

### 🔄 V0.10 命名

**不叫** DAG Executor，**改名** Workflow Runtime：

> 如果未来加入条件分支、重试策略、人工确认、暂停恢复等能力，"Workflow Runtime" 会比 "DAG Executor" 更能覆盖长期演进方向，同时不会限制内部实现。
