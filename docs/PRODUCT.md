# PRODUCT.md
## AI Hub — 多平台 AI 能力聚合入口

> 版本：v0.1 产品定义
> 日期：2026-07-10
> 状态：规划中

---

## 一句话定义

**一个能够自动调用多个 AI 工具、优先使用免费额度完成任务的统一入口。**

---

## 为什么做？

### 痛点

2026 年，个人开发者手上有大量 AI 工具：

- QODER（免费编程）
- Gemini CLI（免费对话）
- ChatGPT（有限免费额度）
- QClaw（腾讯生态）
- Coze（工作流平台）
- Marvis（系统操作）
- Trae Work（并行执行）

**问题不是没有工具，而是工具太多。**

每次完成任务，用户要手动判断：
1. 这个任务该用哪个工具？
2. 这个工具的免费额度还有多少？
3. 额度用完了，换哪个？
4. 不同工具的结果格式不统一，怎么对比？

**用户的真实行为**：大部分人不关心用哪个 AI，只关心任务有没有完成、有没有花自己的钱。

### 解决方案

做一个统一入口：
- 用户只说一句话
- 系统自动选最合适的免费平台
- 统一返回结果
- 用户不需要知道背后是谁干的

---

## 用户是谁？

### 核心用户

**个人开发者和独立工作者**，符合以下特征：

| 特征 | 描述 |
|------|------|
| AI 重度用户 | 每天使用 3 个以上 AI 工具 |
| 成本敏感 | 优先使用免费额度，不愿付费订阅多个平台 |
| 效率导向 | 不想花时间判断"这个任务该用哪个工具" |
| 技术能力 | 能安装 CLI 工具、能编辑配置文件 |

### 非目标用户（第一版不服务）

- 企业团队（需要权限管理、审计日志、SLA）
- 非技术人员（需要 GUI、引导式交互）
- 只用一个 AI 工具就满足的人（没有聚合需求）

---

## 第一版解决什么？

### 核心价值

> **打开一个入口，说一句话，免费把事办了。**

### 具体场景

| 场景 | 用户输入 | 系统行为 |
|------|---------|---------|
| 写代码 | "写一个 Python HTTP 服务" | → QODER 执行 |
| 总结文档 | "总结这个 PDF" | → QClaw 执行 |
| 搜索信息 | "查一下 Rust 1.80 的新特性" | → Gemini CLI 执行 |
| 额度切换 | （QODER 额度用完） | 自动切到 Gemini CLI |
| 统一查看 | "我之前让你写的代码呢" | 从历史记录中取出 |

### 第一版功能清单（5 个）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **Provider 管理** | 显示已接入的 AI 平台列表、在线状态、登录状态、免费额度 | P0 |
| **Router** | 根据任务类型 + 额度 + 优先级，自动选择最合适的 Provider | P0 |
| **统一聊天** | 用户在一个入口输入任务，无论哪个 Provider 执行，结果统一显示 | P0 |
| **历史记录** | 记录每个任务的输入、执行 Provider、输出结果、时间戳 | P1 |
| **额度管理** | 手动或自动维护各平台免费额度剩余，Router 据此做决策 | P1 |

---

## 第一版不做什么

以下功能**坚决不做**，留到后续版本：

| 不做的事 | 原因 | 什么时候做 |
|---------|------|-----------|
| ❌ Agent 编排 | 第一版是 One Task → One Provider，不拆任务 | V0.5+ |
| ❌ 任务分解 | 用户说"做网站"不会自动拆成搜索+编码+部署 | V0.5+ |
| ❌ 多 Agent 并行 | 不需要同时调用多个平台 | V1.0+ |
| ❌ Workflow / 工作流 | 不做可复用流程编排 | V1.0+ |
| ❌ Memory / 记忆系统 | 不做跨会话上下文记忆 | V1.0+ |
| ❌ Vector DB | 不做向量检索 | V2.0+ |
| ❌ 飞书集成 | 不做任何外部协作平台对接 | V1.0+ |
| ❌ MCP 协议 | 不做 Model Context Protocol 支持 | V2.0+ |
| ❌ 插件市场 | 不做第三方插件分发 | V2.0+ |
| ❌ AI 智能路由 | 第一版用规则路由（if-else），不用 AI 做路由决策 | V0.3+ |

**核心原则**：第一版只做"聚合 + 路由 + 统一输出"，不做任何"智能"。

---

## 成功标准

第一版（V0.1）发布后，满足以下条件即为成功：

| 指标 | 目标 |
|------|------|
| 接入 Provider 数量 | ≥ 3 个 |
| 用户完成一个任务的操作步数 | ≤ 2 步（输入 + 获取结果） |
| 路由准确率（选对了平台） | ≥ 80% |
| 免费额度利用率 | 用户不需要手动查额度 |
| 安装到首次使用时间 | ≤ 10 分钟 |

## 风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| Provider 接口变更频繁 | 高 | 能力描述外置为 YAML 配置，接口适配层隔离 |
| 免费额度难以自动检测 | 高 | 第一版支持手动维护，后续逐步自动化 |
| 用户觉得"不如直接用 QODER" | 中 | 价值在于"不需要判断用哪个"，不是"比单个更好" |
| 登录状态管理复杂 | 中 | 第一版要求用户自行登录各平台，系统只检测是否已登录 |
