## V0.1 — AI 聚合器（MVP）

> **目标：一个能自动选平台的 AI 统一入口**
> 预计：2 周
> 核心价值：聚合 + 规则路由 + 免费使用

### 功能清单

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| Provider 管理 | 接入 3 个平台 | QODER、Gemini CLI、QClaw 可用 |
| 规则路由 | 关键词匹配 + 优先级选择 | 输入"写代码"→ QODER，输入"搜索"→ Gemini |
| 统一 CLI | 一个命令完成所有任务 | `ai-hub "任务描述"` 即可执行 |
| 统一结果 | 无论哪个平台，结果格式一致 | Result JSON 格式统一 |
| 历史记录 | 任务记录持久化 | `ai-hub history` 可查最近 10 条 |
| Fallback | 首选不可用自动切换 | QODER 不可用 → Gemini CLI |

### 不做

- ❌ AI 智能路由（用规则 if-else）
- ❌ 额度自动检测（手动维护）
- ❌ 任务分解
- ❌ Web UI
- ❌ 多 Agent

### 里程碑

| 时间点 | 交付物 |
|--------|--------|
| 第 1 天 | 项目结构 + Provider 接口定义 + 配置文件格式 |
| 第 3 天 | 第一个 Provider（QODER）接入完成 |
| 第 5 天 | 第二个 Provider（Gemini CLI）接入完成 |
| 第 7 天 | 第三个 Provider（QClaw）接入完成 |
| 第 10 天 | Router + Fallback + 历史记录完成 |
| 第 12 天 | 端到端测试 + README 文档 |
| 第 14 天 | 发布 V0.1 |

### 开发顺序（严格按此执行）

```
1. 定义 Provider 接口（PROVIDER_SPEC.md 已完成）
2. 实现 Provider 基类 + 配置加载
3. 接入第一个 Provider：QODER
   → 跑通：ai-hub "写代码" → QODER → 返回结果
4. 接入第二个 Provider：Gemini CLI
   → 跑通：ai-hub "搜索" → Gemini → 返回结果
5. 实现 Router（规则路由）
   → 跑通：自动选择 Provider
6. 接入第三个 Provider：QClaw
   → 跑通：ai-hub "分析" → QClaw → 返回结果
7. 实现 Fallback 链
   → 跑通：QODER 不可用 → 自动切 Gemini
8. 实现历史记录
   → 跑通：ai-hub history
9. 端到端测试 20 个场景
10. 写 README + 发布
```

## V0.2 — 额度管理 + Web UI

> **目标：额度可视化管理 + 降低使用门槛**
> 预计：2 周
> 核心价值：额度不愁 + 非开发者可用

### 功能清单

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| 额度看板 | CLI 显示各平台额度状态 | `ai-hub quota` 显示所有平台额度 |

### 不做

## V0.3 — 智能路由

> **目标：用 AI 替代规则做路由决策**
> 预计：3 周
> 核心价值：路由更准、支持模糊意图

### 功能清单

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| AI 任务分类 | 用 LLM 识别任务类型 | 输入"帮我搞个网站"→ 识别为 coding |
| AI Provider 选择 | 用 LLM 综合判断选哪个 | 考虑任务类型 + 额度 + 历史成功率 |
| 规则路由保留 | AI 路由失败时降级为规则 | AI 不可用时回退到 V0.1 的规则路由 |
| 路由解释 | 显示为什么选了这个 Provider | `ai-hub --explain "写代码"` 输出路由理由 |
| 路由准确率统计 | 记录路由是否选对了 | `ai-hub stats` 显示路由准确率 |

### 不做

- ❌ 任务分解（还是 One Task → One Provider）
- ❌ 多 Agent 并行

### 升级策略

```
V0.1 的 Router：
  if task contains "写代码" → QODER

V0.3 的 Router：
  1. 调用 LLM 分类任务 → "coding"
  2. 调用 LLM 选择 Provider → "qoder"（基于能力+额度+历史）
  3. 如果 LLM 不可用 → 降级为 V0.1 的规则路由
```

**关键**：Provider 接口不变，只升级 Router 内部实现。

## V0.5 — 任务分解

> **目标：复合任务自动拆解为子任务**
> 预计：4 周
> 核心价值：从"One Task → One Provider"升级为"One Task → Multiple Providers"

### 功能清单

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| Task Splitter | 将复合任务拆解为子任务 | "做网站"→ 搜索+编码+部署 |
| 子任务路由 | 每个子任务独立路由 | 搜索→QClaw，编码→QODER |
| 依赖管理 | 标注子任务间依赖关系 | T2 依赖 T1 的输出 |
| 串行/并行 | 无依赖的子任务并行执行 | 两个独立搜索同时进行 |
| 结果整合 | 合并多个子任务的结果 | 最终输出整合后的完整结果 |

### 不做

- ❌ Agent 协议（子任务还是通过 Provider 接口执行）
- ❌ 飞书交付
- ❌ Memory

### 架构变化

```
V0.3：
  User → Router → Provider → Result

V0.5：
  User → Splitter → [SubTask1 → Router → Provider → Result1]
                   [SubTask2 → Router → Provider → Result2]
                   [SubTask3 → Router → Provider → Result3]
         → Merger → Final Result
```

**关键**：Provider 接口不变。Splitter 和 Merger 是新增层。

## V1.0 — Agent 编排

> **目标：真正的多 Agent 协同系统**
> 预计：6 周
> 核心价值：WorkBuddy 作为中央大脑，编排所有 Agent

### 功能清单

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| Planner | 完整的需求理解 + 任务分解 | v2.0 提示词模板启用 |
| Orchestrator | 多 Agent 状态追踪 + 结果回收 | 任务看板实时更新 |
| 飞书交付 | Lark CLI 集成 | 结果推送到飞书消息/文档/表格 |
| Memory | 跨会话上下文记忆 | "上次写的代码"能找到 |
| Context Store | 子任务间共享上下文 | T2 能拿到 T1 的输出 |
| Agent 协议 | 下游 Agent 接收/回报模板 | v2.0 的 3.2 节启用 |
| 冲突解决 | 多 Agent 结果冲突处理 | 事实冲突取最新，方案冲突问用户 |

### 架构变化

```
V0.5：
  User → Splitter → [SubTasks] → Merger → Result

V1.0：
  User → WorkBuddy(Planner + Orchestrator + Deliver)
           ↓
         [Agent1 ←→ Agent2 ←→ Agent3]
           ↓
         Result + 飞书交付
```

**关键**：Agent 是 Provider 的超集。`Agent.run(task)` 兼容 `Provider.execute(task)`。v2.0 提示词模板完整启用。

## 版本升级触发条件

不是按时间推进，而是按需求触发：

| 当前版本 | 触发升级的条件 | 升级到 |
|---------|--------------|--------|
| V0.1 | 用户抱怨"额度管理太麻烦" | V0.2 |
| V0.1 | 用户抱怨"路由选错了" | V0.3 |
| V0.2 | 用户提出"为什么不能拆任务" | V0.5 |
| V0.3 | 用户提出"能不能多个平台协同" | V0.5 |
| V0.5 | 用户提出"能不能自动推送到飞书" | V1.0 |
| V0.5 | 用户提出"能不能记住上次的结果" | V1.0 |
| V1.0 | 第三方开发者想接入自己的平台 | V2.0 |

**核心原则**：用户不提，就不做。避免过度设计。

## 风险管理

| 风险 | 影响版本 | 应对 |
|------|---------|------|
| Provider 平台封禁第三方调用 | V0.1 | 使用官方 CLI/API，不做逆向 |
| 免费 FAQ 额度政策变更 | V0.2 | 额度配置外置，随时调整 |
| AI 路由成本高于免费额度 | V0.3 | 用本地小模型做路由分类 |
| 任务分解质量不稳定 | V0.5 | 保留手动分解选项 |
| Agent 编排复杂度失控 | V1.0 | 严格控制 Agent 数量（≤ 5） |
| 插件质量参差不齐 | V2.0 | 插件评分 + 官方认证机制 |
